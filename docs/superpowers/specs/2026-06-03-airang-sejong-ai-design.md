# 아이랑 세종 AI MVP Design

## Goal

`아이랑 세종 AI`는 0~36개월 영유아 가정이 오늘의 조건에 맞는 세종시 공공자원을 빠르게 찾도록 돕는 발표용 데모 웹앱이다. 앱은 기존 공공정보를 대체하지 않고, 보호자의 월령, 생활권, 날씨, 수유, 기저귀, 유모차 조건을 해석해 공식 정보 기준으로 이용 가능성이 높은 선택지를 연결한다.

## Scope

MVP는 백엔드 없는 React 웹앱으로 구현한다. 샘플 공공시설 데이터, 공식정보 Q&A 데이터, 비식별 행정 집계 데이터를 앱 내부에 포함하고, 추천 로직은 테스트 가능한 순수 함수로 분리한다.

포함 범위:

- 시민 추천 화면: 월령, 생활권, 외출 시간대, 날씨, 미세먼지, 이동수단, 보호자 수, 수유실, 기저귀대, 유모차 조건을 입력한다.
- 추천 결과: 공식 정보 기준 이용 가능성이 높은 공공자원 3개 내외를 보여준다.
- 추천 카드: 추천 이유, 월령 적합성, 운영시간, 휴관일, 수유실, 기저귀대, 유모차 접근성, 주차, 비용, 공식 링크, 정보 갱신일을 표시한다.
- 공식정보 Q&A 화면: 영유아 외출, 돌봄, 시간제보육, 예방접종, 건강 이상 질문을 분류하고 공식 출처 기반 답변을 보여준다.
- 건강/응급 안전 전환: 의료 진단이나 치료 조언을 하지 않고 위험신호 확인과 보건소, 의료기관, 119 안내로 제한한다.
- 행정 대시보드: 생활권, 월령대, 시간대, 조건별 검색/추천/제보 흐름을 비식별 집계로 보여준다.

초기 제외 범위:

- 회원가입, 가족 프로필 장기 저장, 아이 이름 저장
- 실시간 예약, 결제, 신청 대행
- 실시간 혼잡도 예측
- 민간 상업시설 추천
- 커뮤니티 후기, 별점, 사용자 간 돌봄 매칭
- 의료 진단, 약 복용 안내, 치료법 제시

## Users

주 사용자는 세종시에 거주하거나 세종시 공공자원을 이용하는 0~36개월 영유아 보호자다. 초보 양육자, 맞벌이 가정, 쌍둥이와 다자녀 가정, 전입 가정처럼 지역 육아정보를 직접 조합하기 어려운 사용자를 우선한다.

보조 사용자는 행정 담당자다. 담당자는 개인 추적이 아닌 생활권, 월령대, 조건별 비식별 통계를 통해 반복 수요와 시설 개선 신호를 확인한다.

## Product Flow

### Citizen Recommendation

사용자는 한 화면에서 오늘 조건을 입력한다. 예시는 "8개월 쌍둥이와 비 오는 오후에 갈 수 있는 실내 공간, 쌍둥이 유모차와 수유실 필요"이다.

앱은 입력값을 정규화하고, 필수 조건에 맞지 않는 시설을 제외한 뒤, 다음 기준으로 점수를 계산한다.

- 영유아 편의: 수유실, 기저귀 교환대, 세면과 온수
- 이동 편의: 일반 유모차, 광폭 또는 쌍둥이 유모차, 엘리베이터, 경사로
- 주차 편의: 주차 가능, 실내 주차, 주차 규모
- 혼잡도: 평일, 주말, 시간대별 예상 혼잡
- 비용: 무료, 유료, 예상 비용
- 운영시간: 운영시간, 휴관일, 예약 필요 여부
- 환경 적합도: 우천, 폭염, 미세먼지, 실내외 조건
- 월령 적합도: 0~6개월, 7~12개월, 13~24개월, 25~36개월

결과는 상위 3개 내외로 제한한다. 각 결과는 "조건상 적합해 보이는 이유"와 "방문 전 공식 링크 또는 전화 확인 필요" 안내를 함께 제공한다.

### Official Information Q&A

사용자는 자연어로 질문한다. 앱은 질문을 외출, 보육, 건강, 예방접종, 프로그램, 응급안내 중 하나로 분류한다.

일반 질문은 샘플 공식정보 데이터에서 관련 근거를 찾아 요약한다. 답변에는 출처명, 담당기관, 기준일, 공식 링크를 포함한다.

건강, 응급, 발달 관련 질문은 보수적으로 처리한다. 앱은 정상/비정상 판단이나 치료법을 말하지 않는다. 대신 위험신호 체크리스트와 "이 서비스는 의료 진단이나 치료 조언을 제공하지 않습니다" 문구를 표시하고, 위험신호가 있으면 의료기관 또는 119 안내로 전환한다.

### Administrative Dashboard

대시보드는 개인 데이터를 표시하지 않는다. 앱 내부 샘플 이벤트를 집계해 다음 지표를 보여준다.

- 생활권별 수유실, 기저귀대, 유모차 접근성 수요
- 월령대별 추천 요청 비중
- 시간대별 외출/돌봄 수요
- 대체검색 또는 제보가 많은 시설 조건
- 공식정보 클릭률과 추천 전환 흐름

대시보드에는 "비식별 집계 데이터이며 개인·가구별 이용 여부를 판단하는 용도로 사용할 수 없습니다"라는 안내를 고정 표시한다.

## Architecture

기술 스택은 Vite, React, TypeScript, Vitest, Testing Library를 사용한다. 백엔드는 만들지 않는다. 모든 데이터는 `src/data`에 샘플로 두고, 도메인 로직은 `src/features` 아래 순수 함수로 분리한다.

추천 기능 파일 구조:

- `src/features/recommendation/types.ts`: 시설과 선호 조건 타입
- `src/features/recommendation/normalizePreferences.ts`: 입력값 정규화
- `src/features/recommendation/filterFacilities.ts`: 필수 조건 필터링
- `src/features/recommendation/scoreFacility.ts`: 8개 기준 점수화
- `src/features/recommendation/rankFacilities.ts`: 정렬과 상위 추천 생성
- `src/features/recommendation/explainRecommendation.ts`: 추천 이유 생성

Q&A 기능 파일 구조:

- `src/features/qa/types.ts`: 질문, 출처, 답변 타입
- `src/features/qa/classifyQuestion.ts`: 질문 분류
- `src/features/qa/searchOfficialInfo.ts`: 공식정보 후보 검색
- `src/features/qa/buildAnswer.ts`: 출처 기반 답변 생성과 건강 질문 안전 전환

대시보드 기능 파일 구조:

- `src/features/dashboard/types.ts`: 이벤트와 집계 타입
- `src/features/dashboard/dashboardMetrics.ts`: 비식별 집계 지표 생성

UI는 `src/pages`에 화면 단위로 두고, 반복 컴포넌트는 `src/components`에 둔다. 화면은 라우팅 대신 한 앱 안의 탭 전환으로 구성한다. 이는 발표용 데모에서 사용자가 흐름을 놓치지 않도록 하기 위함이다.

## Data Model

시설 데이터는 실제 공공정보를 닮은 샘플 구조로 구성한다. 각 시설에는 다음 필드를 포함한다.

- 시설명, 유형, 생활권, 주소
- 운영시간, 휴관일, 예약 필요 여부
- 수유실, 기저귀대, 세면/온수 가능 여부
- 일반 유모차, 쌍둥이 유모차, 엘리베이터, 경사로
- 실내 체류 가능성, 우천/폭염/미세먼지 적합도
- 주차, 비용, 전화번호, 공식 링크
- 정보 갱신 기준일과 관리 범위

공식정보 데이터는 출처명, 담당기관, 주제, 월령대, 요약, 공식 링크, 기준일을 포함한다.

행정 이벤트 데이터는 생활권, 월령대, 시간대, 조건 태그, 이벤트 유형만 가진다. 개인 식별자, 상세주소, 아이 이름, 건강기록은 포함하지 않는다.

## Safety And Privacy

추천 문구는 보장형 표현을 피한다. "안전합니다", "반드시 이용 가능" 대신 "공식 정보 기준 이용 가능성이 높습니다", "조건상 적합해 보입니다", "방문 전 공식 링크 또는 전화로 확인하세요"를 사용한다.

건강 질문은 의료 판단으로 이어지지 않게 제한한다. 발열, 구토, 호흡곤란, 경련, 의식 저하, 약 복용, 예방접종 이상반응 같은 단어가 감지되면 의료기관 또는 119 안내를 우선한다.

개인정보는 최소화한다. 앱은 회원가입, 아이 이름, 주민등록번호, 상세주소, 장기 프로필 저장을 제공하지 않는다. 대시보드는 비식별 집계만 사용한다.

## Visual Direction

앱은 행정·복지 서비스 성격에 맞게 차분하고 실용적인 인터페이스로 만든다. 첫 화면은 마케팅 랜딩 페이지가 아니라 바로 사용할 수 있는 추천 입력 화면이다.

시각 톤은 밝은 배경, 명확한 폼, 표와 카드 중심의 정보 구조로 간다. 장식적 히어로나 과한 그라데이션은 사용하지 않는다. 버튼에는 가능한 경우 lucide 아이콘을 사용하고, 카드 반경은 8px 이하로 유지한다.

## Testing

테스트는 로직 중심으로 작성한다.

- 추천 엔진은 월령, 날씨, 미세먼지, 수유실, 기저귀대, 쌍둥이 유모차, 운영시간 조건을 검증한다.
- Q&A는 공식 출처 포함, 건강 질문 안전 전환, 주제 분류를 검증한다.
- 대시보드는 비식별 이벤트 집계와 개인 식별 필드 부재를 검증한다.
- UI는 주요 화면 렌더링, 폼 입력 후 추천 결과 표시, 탭 전환을 검증한다.

최종 검증은 `npm test`, `npm run build`, 로컬 dev server, 브라우저 화면 확인으로 한다.

## Agent Plan

구현은 충돌을 줄이기 위해 하위 에이전트 분업을 사용한다.

- 데이터 에이전트: 샘플 시설, 공식정보, 행정 이벤트 데이터 구조 검토
- 추천 엔진 에이전트: 추천 로직과 테스트 구현
- Q&A 에이전트: 질문 분류, 공식정보 답변, 안전 전환 구현
- 대시보드 에이전트: 비식별 집계 로직 구현
- UI 에이전트: 화면 컴포넌트 조립과 반응형 스타일 검토

공통 타입과 핵심 파일 구조는 먼저 고정하고, 이후 각 에이전트의 write scope를 분리한다. 최종 통합은 본 세션에서 검토하고 테스트한다.

## Success Criteria

- 앱을 로컬에서 실행하면 첫 화면에서 시민 추천 기능을 바로 사용할 수 있다.
- 추천 결과는 3개 내외이며 각 결과에 추천 이유, 공식 링크, 갱신일, 방문 전 확인 안내가 있다.
- 공식정보 Q&A는 출처 기반 답변을 제공하고 건강 질문은 진단 없이 안전 안내로 전환한다.
- 행정 대시보드는 비식별 집계 지표를 보여주며 개인 추적 금지 문구가 있다.
- 추천, Q&A, 대시보드 로직 테스트가 통과한다.
- production build가 성공한다.
- 브라우저에서 데스크톱과 모바일 주요 화면이 깨지지 않는다.
